# 🧠 Deep-Dive: PII Leakage Guard (Outbound Data Loss Prevention)

### 📌 The Problem: The "Memory" of LLMs
Large Language Models (LLMs) are trained on trillions of tokens from the public internet, which often contain sensitive data. Furthermore, in enterprise **RAG (Retrieval-Augmented Generation)** systems, the AI has access to internal databases and proprietary context. 

The core threat is **Non-Deterministic Data Exfiltration**: An AI might accidentally "hallucinate" or recall a real person's email, a private API key, or an internal server IP address when generating a response. If this data reaches the end-user, it constitutes a **GDPR, HIPAA, or SOC2 compliance breach**.

### 🏗️ Architecture: The "Zero-Trust" Outbound Proxy
This project implements an **Outbound Proxy Layer**. It operates on the principle of **Zero Trust** regarding AI output: no string generated by an LLM is considered safe until it has been inspected by a deterministic (rule-based) engine.



#### 1. Deterministic vs. Probabilistic Filtering
While the LLM is **probabilistic** (predicting the next likely word), our Guard is **deterministic** (applying immutable security rules). This provides a "Hard Safety Rail" that cannot be bypassed by prompt engineering or jailbreaking techniques.

#### 2. The Regex Engine (Pattern Recognition)
We utilize optimized **Regular Expressions (Regex)** to identify the unique "digital signatures" of high-risk data:
* **Email Identification**: Captures standard RFC 5322 compliant address structures.
* **Network Obfuscation**: By redacting **IPv4 addresses**, we prevent **Infrastructure Mapping**, where an attacker attempts to learn the internal IP topography of the cloud environment.
* **Credential Protection**: We target high-entropy strings like **API Keys** (e.g., `sk-...`) that represent a direct path to system compromise.
* **Financial Data**: Scans for 13-16 digit sequences matching Credit Card structures.

### 🛡️ Implementation: The Redaction Workflow
When the AI generates a response, the PII Guard intercepts the buffer before it is rendered to the user:

| Phase | Action | Technical Logic |
| :--- | :--- | :--- |
| **Ingestion** | `Intercept` | Receives the raw completion string from the LLM provider API. |
| **Scanning** | `Match` | Parallel execution of Regex patterns against the string buffer. |
| **Redaction** | `Replace` | Substitutes sensitive substrings with safe labels (e.g., `[EMAIL_REDACTED]`). |
| **Auditing** | `Notify` | Generates a **High-Severity Warning** in the security logs for human review. |

### 🧪 Defense-in-Depth: Closing the Loop
This project completes the **Adversarial AI Lab** by securing both ends of the AI communication pipe:
1.  **Project 01 (Prompt Firewall)**: Sanitizes instructions entering the model (**Input Security**).
2.  **Project 02 (PII Guard)**: Sanitizes data leaving the model (**Output Security**).

This dual-layered approach creates a "Sanitized Sandbox," ensuring that even if an attacker successfully bypasses the input filter with a novel "Jailbreak," the sensitive data is caught at the final exit point.
